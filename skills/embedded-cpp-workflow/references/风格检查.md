# 风格检查

## 自查清单

完成代码风格相关改动前检查：

- 是否使用 `snake_case`，包括类名、结构体名、枚举名、函数名和变量名。
- 结构体、枚举、普通类型是否没有 `_t` 后缀。
- 整数类型是否使用 `int16_t`、`uint32_t` 等显式定宽类型。
- 常量是否使用 `UPPER_SNAKE_CASE`。
- 缩进是否为 4 空格，且每层作用域都有缩进。
- 函数、`namespace`、`class`、`struct`、`enum` 左括号是否换行。
- 单行初始化是否保持在一行。
- 多行数组、结构体或列表初始化是否避免 `= {` 后换行。
- 位运算和移位是否有空格，例如 `1 << 2`。
- 函数内未使用参数是否没有写 `(void)arg;` 这类占位语句。
- `.cpp` 第一个 include 是否是同名头文件，且后面空一行再 include 项目依赖。
- `.h` namespace 中函数顺序是否为 API、`init()`、RTOS 任务入口。
- 较长函数声明换行时，续行是否相对首行增加 4 空格缩进且没有 tab。
- `.cpp` 中 `static` helper 是否在对外 API 前，对外 API 顺序是否尽量与 `.h` 一致。
- `.cpp` 中是否没有使用 namespace 别名或 `using` 简写替代完整限定名。
- `.cpp` 中定义的普通函数和 `static` helper 是否都有中文 Doxygen 注释；局部类成员函数可不逐个注释。
- 是否区分项目代码和公共库/框架/SDK API，没有为了风格强行改第三方命名、宏名或回调签名。
- 是否没有顺手改硬件参数、协议格式、任务参数或控制语义。

## 可以运行的检查

- 如果仓库是 git 项目，运行 `git diff --check` 检查空白问题。
- 如果改动影响头文件、函数签名或 include，尽量运行构建。
- PlatformIO 项目优先在含 `platformio.ini` 的目录运行构建命令。
- 如果格式化工具会破坏本规范，不要直接全量运行。

## 汇报方式

回答用户时区分：

- 纯风格改动。
- 风格顺带的小范围结构整理。
- 可能影响行为的改动。
- 已验证内容和未验证内容。
