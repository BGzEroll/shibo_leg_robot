# 注释与可读性

## 注释总则

- `.cpp` 中定义的函数都写中文 Doxygen 注释，包括简单 getter。
- `.h` 中的函数声明可以不写注释；只有接口语义不明显、参数含义容易误解时才补注释。
- `static` 私有 helper 也写注释。
- `.cpp` 内部实现用的局部类，其成员函数可以不逐个写 Doxygen 注释；类的职责或不直观逻辑需要用类级注释或少量关键注释说明。
- 没有参数或返回值时，不要硬写空的 `@param` 或 `@return`。

函数注释格式：

```cpp
/**
 * @brief 解析 MaixCam 视觉测量帧
 *
 * @param frame 接收帧缓冲区
 *
 * @return 解析成功时返回 true
 */
static bool parse_vision(uint8_t *frame)
{
    return true;
}
```

无返回值示例：

```cpp
/**
 * @brief 初始化控制器模块及其内部子模块
 */
void controller::init()
{
}
```

简单 getter 也写注释：

```cpp
/**
 * @brief 获取上位机遥控输入队列
 *
 * @return 队列句柄
 */
QueueHandle_t host_comm::remote_queue()
{
    return rx_queue;
}
```

`.cpp` 内局部类可以简化：

```cpp
// 管理一个 UART 实例及其初始化状态。
class uart_dev
{
    public:
        void init()
        {
            if(is_init || !serial){return;}
            is_init = true;
            serial->begin(baudrate);
        }
};
```

## 需要重点说明的内容

- 协议帧格式、字节宽度、大小端、校验方式。
- 硬件引脚、角度范围、限幅、单位换算。
- RTOS 任务职责、周期、优先级选择。
- 状态机阶段和跳转条件。
- 控制算法里不直观的参数、滤波、死区、积分重置。
- 会影响调试或安全的阈值。

## 不需要注释的地方

- 局部变量的简单赋值。
- `i++`、`return true` 这类语句本身。
- 为了显得完整而补的空泛说明。

## 可读性

- helper 名字要说明它为什么存在，例如 `consume_middle_calibration_request()` 比 `handle_request()` 更好。
- 不为了抽象而抽象。只有当 helper 降低复杂度、隔离硬件细节或表达状态机阶段时才拆。
- 魔法数字应改成有名字的常量；不知道含义时先保留或询问，不要乱命名。
- 复杂表达式可以拆成中间变量，中间变量名要体现单位、方向、协议字段或物理意义。
